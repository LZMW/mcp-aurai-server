## 1. 项目定位

这个 MCP 的作用可以理解成：

“本地 AI 遇到复杂问题时，再找一个更高层的顾问继续分析。”

它不是替代本地 AI，而是给本地 AI 多一个升级求助通道。典型流程是：

1. 本地 AI 发现问题复杂
2. 调用 `consult_aurai`
3. 如果资料不够，再调用 `sync_context`
4. 执行建议后，用 `report_progress` 汇报结果
5. 继续多轮迭代，直到问题解决

## 2. 安装方法

更详细的步骤见：

- [Claude Code 安装指南](CLAUDE_CODE_INSTALL.md)
- [README](../README.md)

这里给一份够用的标准流程。

### 2.1 安装依赖

```bash
cd G:\codex\mcp-aurai-server
python -m venv venv
venv\Scripts\activate
pip install -e ".[all-dev]"
```

### 2.2 注册到 Claude Code

```bash
claude mcp add --scope user --transport stdio aurai-advisor ^
  --env AURAI_API_KEY="your-api-key" ^
  --env AURAI_BASE_URL="https://api.example.com/v1" ^
  --env AURAI_MODEL="gpt-4o" ^
  -- "G:\codex\mcp-aurai-server\venv\Scripts\python.exe" "-m" "mcp_aurai.server"
```

### 2.3 验证

```bash
claude mcp list
pytest
```

如果这两步都正常，说明安装基本完成。

## 3. 配置项

### 3.1 必填

| 变量 | 说明 |
|------|------|
| `AURAI_API_KEY` | API 密钥 |
| `AURAI_BASE_URL` | OpenAI 兼容接口地址 |
| `AURAI_MODEL` | 模型名称 |

注意：

- 当前版本只走 OpenAI 兼容接口
- 不再依赖旧版的 `AURAI_PROVIDER`

### 3.2 常用可选项

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `AURAI_TEMPERATURE` | 温度 | `0.7` |
| `AURAI_MAX_ITERATIONS` | 最大迭代轮数 | `10` |
| `AURAI_MAX_HISTORY` | 每个会话最多保留多少条历史 | `50` |
| `AURAI_CONTEXT_WINDOW` | 总上下文窗口 | `200000` |
| `AURAI_MAX_MESSAGE_TOKENS` | 单条大文件消息上限 | `150000` |
| `AURAI_MAX_TOKENS` | 最大输出长度 | `32000` |
| `AURAI_LOG_LEVEL` | 日志级别 | `INFO` |
| `AURAI_ENABLE_PERSISTENCE` | 是否写入历史文件 | `true` |
| `AURAI_HISTORY_PATH` | 默认会话历史文件路径 | `~/.mcp-aurai/history.json` |
| `AURAI_HISTORY_LOCK_TIMEOUT` | 历史文件锁等待时间（秒） | `10` |
| `AURAI_ENABLE_HISTORY_SUMMARY` | 是否启用历史摘要 | `true` |
| `AURAI_HISTORY_SUMMARY_KEEP_RECENT` | 摘要后保留最近几轮原始记录 | `3` |
| `AURAI_HISTORY_SUMMARY_TRIGGER` | 达到多少原始记录后触发摘要 | `8` |

### 3.3 推荐配置示例

```bash
AURAI_API_KEY=your-api-key
AURAI_BASE_URL=https://open.bigmodel.cn/api/paas/v4/
AURAI_MODEL=glm-4.7

AURAI_MAX_ITERATIONS=10
AURAI_CONTEXT_WINDOW=200000
AURAI_MAX_MESSAGE_TOKENS=150000
AURAI_MAX_TOKENS=32000
AURAI_LOG_LEVEL=INFO
```

## 4. 工具说明

### 4.1 `consult_aurai`

用途：向上级顾问发起问题咨询。

常用参数：

- `problem_type`
- `error_message`
- `code_snippet`
- `context`
- `attempts_made`
- `answers_to_questions`
- `is_new_question`
- `session_id`

典型示例：

```python
consult_aurai(
    problem_type="runtime_error",
    error_message="启动时报 KeyError: api_key",
    code_snippet="config = load_config()\napi_key = config['api_key']",
    context={
        "file_path": "src/config.py",
        "terminal_output": "Traceback ..."
    },
    session_id="startup-error"
)
```

### 4.2 `sync_context`

用途：把文件和项目背景同步给上级顾问。

参数：

- `operation`: `full_sync` / `incremental` / `clear`
- `files`
- `project_info`
- `session_id`

重点：

- 现在不再要求手动把代码文件复制成 `.txt`
- 文本代码/配置文件会自动转换后发送
- 二进制文件会跳过

### 4.3 `report_progress`

用途：执行建议后回报结果，让上级顾问继续给下一步。

参数：

- `actions_taken`
- `result`
- `new_error`
- `feedback`
- `session_id`

### 4.4 `get_status`

用途：查看当前会话状态。

返回内容通常包括：

- `session_id`
- `conversation_history_count`
- `history_path`
- `max_iterations`
- `max_history`
- `provider`
- `model`

## 5. 文件同步规则

### 5.1 什么文件会被发送

会直接或自动转换后发送的，核心条件是：

“文件内容是文本，且能被当作文本读出来。”

例如：

- `README.md`
- `notes.txt`
- `main.py`
- `app.js`
- `config.json`
- `docker-compose.yml`
- `.env`
- `settings.toml`

### 5.2 什么文件会被跳过

这些一般会被视为二进制：

- 图片
- 压缩包
- 音视频
- 可执行文件
- 明显含二进制内容的文件

### 5.3 自动转换是什么意思

比如你上传：

```python
sync_context(
    operation="incremental",
    files=["src/main.py", "config/settings.json"]
)
```

系统会自动：

1. 读取这两个文本文件
2. 给它们生成适合发给上级顾问的文本发送名
3. 在内容前补上原始路径说明
4. 作为文本上下文发送给上级顾问

你可以从返回结果里的这些字段看到详情：

- `uploaded_files`
- `auto_converted_files`
- `skipped_files`

## 6. 会话与历史管理

### 6.1 `session_id`

如果你同时处理多个问题，最推荐给每个问题一个单独的 `session_id`。

例如：

```python
consult_aurai(
    problem_type="runtime_error",
    error_message="问题 A",
    session_id="issue-a"
)

consult_aurai(
    problem_type="design_issue",
    error_message="问题 B",
    session_id="issue-b"
)
```

实际效果：

- 问题 A 和问题 B 各有自己的“笔记本”
- 不会因为一个问题的旧上下文污染另一个问题

### 6.2 自动清空

会触发清空历史的常见情况：

- 上级顾问返回 `resolved=true`
- 你主动设置 `is_new_question=true`
- 你调用 `sync_context(operation="clear")`

### 6.3 历史摘要

历史记录太长时，系统会把较早内容压成一条摘要。

可以把它理解成：

“旧聊天记录被整理成会议纪要，最近几轮原文仍保留。”

这样做的好处：

- 节省上下文空间
- 降低模型重复阅读旧对话的成本
- 给当前问题和新上传文件腾位置

### 6.4 历史文件更稳

当前版本对历史文件做了两层保护：

1. 锁文件
   防止多个进程同时改同一个历史文件

2. 原子写入
   先写临时文件，再整块替换正式文件，减少留下半截 JSON 的风险

## 7. 常见使用方式

### 7.1 直接提问

适合中小问题：

```python
consult_aurai(
    problem_type="runtime_error",
    error_message="单元测试失败"
)
```

### 7.2 先同步文件再提问

适合大型问题：

```python
sync_context(
    operation="full_sync",
    files=["README.md", "src/main.py", "src/router.ts", "config/app.json"],
    project_info={
        "project_name": "Example",
        "tech_stack": "Python + React"
    },
    session_id="project-debug"
)

consult_aurai(
    problem_type="runtime_error",
    error_message="请结合已同步文件排查启动失败",
    session_id="project-debug"
)
```

### 7.3 多轮迭代

```python
consult_aurai(...)
report_progress(...)
report_progress(...)
```

这适合那些“一次建议不够，需要边试边推进”的问题。

## 8. 故障排查

### 8.1 安装后看不到 MCP

```bash
claude mcp list
claude mcp remove aurai-advisor -s user
claude mcp add --scope user ...
```

### 8.2 报 `ModuleNotFoundError`

一般是虚拟环境没装好：

```bash
cd G:\codex\mcp-aurai-server
python -m venv venv
venv\Scripts\activate
pip install -e ".[all-dev]"
```

### 8.3 上级顾问没收到我传的文件

优先看 `sync_context` 返回结果：

- `uploaded_files`
- `auto_converted_files`
- `skipped_files`

最常见原因：

- 文件路径不存在
- 文件是二进制
- 文件内容本身无法当作文本读取

### 8.4 不同问题互相影响

给它们传不同的 `session_id`。

### 8.5 为什么历史数量突然变少了

这通常是历史摘要生效了，不是数据丢了，而是旧记录被压成摘要了。
